# 移动端适配计划（React + Vite + Ant Design）

版本：v1.0  
日期：2026-03-17  
适用项目：React 18 + TypeScript + Vite + Ant Design + Less/CSS Modules + React Router + Zustand

## 1. 目标与原则

### 1.1 目标

- 在常见手机宽度（320–430px）下页面**可用、好用、可读**，交互不“挤”、不误触
- 平板/桌面端自然扩展，不出现明显布局抖动或信息错位
- 不引入明显性能回退（首屏渲染、滚动、弹窗交互保持流畅）

### 1.2 原则（强制约束）

- **Mobile First**：先保证 `xs`（小屏）体验，再向上增强
- **布局优先使用 flex**，禁止使用 `float`
- **样式模块化优先**：使用 CSS Modules（`.module.less`），避免全局样式污染
- **安全**：禁止使用 `eval()`、`with()` 等不安全函数
- **前缀自动补齐**：依赖构建链路（PostCSS/Autoprefixer）自动添加浏览器前缀，避免手写散落

## 2. 适配范围（页面与关键模块）

- `home-page`：导航/分类展示/网站卡片/搜索框
- `manage-page`：编辑表单/列表或表格/弹窗或抽屉
- `about-page`
- 公共组件：`layout`（Header/导航/Footer）、`web-item`、`search-box` 等

## 3. 统一规范（断点、布局、尺寸、交互）

### 3.1 断点体系（与 Ant Design 对齐）

建议统一使用以下断点口径，减少组件间“各写各的”：

- `xs`：`< 576px`
- `sm`：`≥ 576px`
- `md`：`≥ 768px`
- `lg`：`≥ 992px`
- `xl`：`≥ 1200px`

### 3.2 布局策略（建议选型）

- **页面骨架**：`xs` 下尽量单列；侧边导航折叠为 `Drawer`；`sm+` 再恢复多列布局
- **列表/卡片**：推荐响应式网格
  - `xs`：1 列
  - `sm`：2 列
  - `md`：3 列（按内容密度微调）
- **文本溢出**：标题/描述统一做 1–2 行截断或折叠，保证首屏可扫读

### 3.3 尺寸与排版

- 推荐 `rem/em` + 断点控制；必要时用 `clamp()` 实现“随屏幕平滑变化”
- 触控友好：主要按钮/可点击区域建议 ≥ 44px（高度或点击热区）

### 3.4 刘海屏与安全区

- 关键区域（Footer、底部操作栏、Drawer 底部）预留 `safe-area-inset-bottom`
- 顶部沉浸式区域（如有）预留 `safe-area-inset-top`

## 4. 实施步骤（按优先级）

> 说明：工期为经验值，可根据实际页面复杂度微调。

### 4.1 现状审计（0.5 天）

- 使用 Chrome DevTools 设备模拟，至少覆盖：
  - `320×568`、`375×812`、`390×844`、`430×932`
- 输出问题清单（建议按严重程度分级）：
  - 横向溢出/出现横向滚动条
  - 内容遮挡（Header/Footer/弹窗遮住正文）
  - 点击区域过小/误触
  - 信息密度过高（不可读、无法扫读）
  - 输入法顶起导致按钮不可达

### 4.2 基础设施改造（0.5 天）

- 校验 `meta viewport` 配置正确（避免按桌面宽度缩放）
- 建立统一的响应式“设计变量”（Less 变量或 `:root` 变量）：
  - 字号、间距、圆角、容器边距、内容最大宽度等
- 增加少量通用能力（谨慎全局化）：
  - 容器统一 padding
  - 文本截断（单行/多行）
  - 安全区 padding（iOS）

### 4.3 Layout 与导航适配（1 天）

- `xs`：
  - 导航改为 `Drawer` 或折叠菜单，Header 更紧凑（标题可缩略）
  - 打开抽屉时防止滚动穿透
- Footer：
  - 小屏降低信息密度，必要内容保留，其余折叠/收起
- 路由切换体验：
  - 页面切换回到顶部（或按业务保留滚动位置，二选一保持一致）

### 4.4 Home 页适配（1 天）

- 搜索框：全宽显示，按钮与图标尺寸适配触控
- 分类/网站卡片：响应式网格；避免固定高度导致内容溢出
- 文本展示：标题 1–2 行截断；次要信息可折叠/省略，保证首屏阅读效率

### 4.5 Manage 页适配（1–2 天）

> 管理页通常是移动端适配成本最高的页面，建议优先处理可用性。

- 列表/表格：
  - `xs` 下优先将 `Table` 方案降级为 Card List（每条记录一张卡）
  - 若必须保留表格，给表格容器提供可控横向滚动，并保证关键列可见/可操作
- 表单：
  - 小屏建议 `label` 顶部布局（更利于纵向滚动与输入）
  - 操作按钮纵向堆叠，或提供吸底操作栏（注意安全区）
- 弹窗：
  - 小屏优先 `Drawer`（近全屏/全屏），避免 `Modal` 高度超出导致按钮不可达

### 4.6 样式收口与质量（0.5 天）

- 统一间距 token，清理无必要的固定宽度/高度
- 确认布局均使用 flex（禁止 `float`）
- 代码质量检查：
  - `pnpm run lint`
  - `pnpm run format:check`

## 5. 测试与验收标准（上线前必过）

### 5.1 视觉与布局

- 在 `320px` 宽度下页面无横向滚动（除非明确允许的局部区域，如表格容器）
- 首屏信息可读（不会“挤成一团”），关键 CTA（主要按钮）可见
- 不出现明显布局抖动（尤其是抽屉/弹窗打开关闭、路由切换）

### 5.2 交互可用性

- 主要按钮/链接易点，不出现频繁误触
- 输入框聚焦时，底部按钮不被输入法遮挡（必要时页面自动滚动到可见区域）
- 抽屉/弹窗内容在小屏可完整操作（可滚动、底部按钮可触达）

### 5.3 兼容性

- iOS Safari + Android Chrome 基本可用
- 刘海屏安全区不遮挡底部操作（含 Footer、吸底按钮、Drawer 底部）
